# 买家角色自动绑定设计

日期：`2026-04-25`

## 目标

完成阶段 3 的第一步角色语义化工作：

- 普通注册用户默认绑定 `buyer` 角色
- 创作者继续通过审核追加 `creator`
- `admin` 继续表示平台管理员，不新增 `platform_admin`
- 为现有 `common` 用户补绑 `buyer`，避免历史数据与新规则脱节

## 当前现状

- 系统初始化只有 `admin` 和 `common` 两个角色
- `SysRegisterService` 当前只创建 `sys_user` 记录，不分配任何业务角色
- `creative` 业务已经在审核通过时追加 `creator`，但逻辑仍分散在创作者服务内部
- 当前项目还没有前台 `/portal`，注册用户如果没有菜单授权，登录后几乎无法使用系统

## 方案比较

### 方案 A：继续复用 `common`

- 优点：改动最小
- 缺点：业务角色和脚手架默认角色混在一起，后续数据权限和论文叙述都会变脏

### 方案 B：新增 `buyer`，注册自动绑定

- 优点：角色语义清晰，后续“买家只看本人数据”能直接基于业务角色收口
- 优点：与 `creator` 审核追加模型一致，边界清楚
- 缺点：需要补注册链路和历史数据迁移

### 方案 C：首次业务行为再懒分配 `buyer`

- 优点：不改注册入口
- 缺点：规则分散在需求、订单、收藏等多个入口，后续排查困难

**结论：采用方案 B。**

## 设计

### 角色模型

- `admin`：继续视为平台管理员
- `buyer`：普通注册用户默认业务角色
- `creator`：创作者审核通过后追加
- `common`：保留历史兼容，不再作为后续业务权限的依据

### 注册链路

- 保持 `SysRegisterService` 负责注册校验与调用入口
- 把默认角色追加落到 `ISysUserService.registerUser` 内部，避免框架层直接操作角色表
- `registerUser` 成功写入用户后，追加 `buyer` 角色
- 如果 `buyer` 角色不存在，直接抛出 `ServiceException`，借助事务回滚用户插入，避免“有账号没业务角色”的脏状态

### 角色追加规则

- 在 `ISysUserService` 中新增通用能力：按 `roleKey` 追加角色，且避免重复插入
- 注册绑定 `buyer` 与审核绑定 `creator` 统一复用该方法
- 如果用户已拥有目标角色，直接跳过

### SQL 初始化与迁移

- 新增增量脚本初始化 `buyer` 角色
- 角色属性先与 `common` 对齐，复制其 `role_menu` 和 `role_dept` 授权，保证当前阶段注册用户仍能进入系统
- 为已有绑定 `common` 的用户补绑 `buyer`
- 不删除 `common`，不替换原有关联，只做补充

## 风险与取舍

- 当前给 `buyer` 复制 `common` 菜单授权是过渡策略，不是最终权限模型
- 真正的“只看本人数据”放到阶段 3 后续数据权限任务中处理
- 若后续前台 `/portal` 上线，可以再逐步收紧 `buyer` 的后台菜单权限

## 测试策略

- `registerUser` 成功时会自动追加 `buyer`
- 目标角色已存在时不重复写入
- 缺少 `buyer` 角色时抛出明确异常
- 创作者审核通过改为复用通用角色追加能力，确保 `buyer + creator` 可共存
