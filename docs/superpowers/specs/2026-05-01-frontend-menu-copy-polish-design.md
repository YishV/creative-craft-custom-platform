# 前端菜单图标与文案补全设计

日期：2026-05-01

## 1. 背景

当前项目前端页面整体已具备演示闭环，但导航菜单、`/portal` 首页快捷入口、主要按钮与空状态文案存在以下问题：

- 个别图标语义重复或不够贴切；
- 页面间文案风格不统一，部分提示语偏口语化或偏占位；
- 同一业务在不同入口中的标题、按钮、提示语不完全一致。

本次工作目标不是重做视觉设计，而是在不改业务逻辑的前提下，补齐缺失项并统一表达。

## 2. 范围

仅处理前端可见层，包含：

- 路由常量中的菜单标题与图标；
- `/portal` 首页快捷入口卡片文案与图标；
- `/portal` 相关页面的主标题、主要操作按钮、返回入口；
- 空状态、占位说明、成功提示等可见文案；
- 必要时同步调整少量后台文创页面的按钮/提示语，使表达一致。

明确不包含：

- 后端菜单 SQL、权限点、接口定义；
- 业务状态机、字段语义、接口入参出参；
- 全站视觉重构或组件替换。

## 3. 目标

完成后应满足：

- 菜单标题清晰、无缺字、无明显错词；
- 菜单与快捷入口图标语义基本匹配，避免明显重复或误导；
- 主要按钮和空状态文案表达自然、简洁、统一；
- 不改变现有业务流程和页面跳转；
- 前端生产构建通过。

## 4. 设计原则

### 4.1 图标补全原则

- 优先使用项目当前已在用的 Element UI / RuoYi 图标体系；
- 同一类导航尽量保持语义一致，例如交易相关使用订单/支付/购物语义，社区相关使用评论/消息/用户语义；
- 仅在存在明显不贴切、冲突或空缺时调整，不为了“更好看”大面积替换。

### 4.2 文案补全原则

- 统一为简洁、偏产品化的中文；
- 保留毕设演示属性，但移除过于随意、调侃式表达；
- 同一动作在不同页面优先使用同一叫法，例如“去结算”“确认支付”“查看详情”“返回列表”；
- 空状态文案先说明“当前没有数据”，再给出下一步操作方向。

### 4.3 改动控制原则

- 优先修改路由元信息和页面本地文案；
- 不改变接口调用、不新增状态字段；
- 单文件改动尽量集中在文案和图标字段，避免掺入无关重构。

## 5. 执行方案

### 5.1 扫描

先扫描以下文件区域：

- `ruoyi-ui/src/router/index.js`
- `ruoyi-ui/src/views/portal/*.vue`
- `ruoyi-ui/src/views/creative/**/*.vue`

重点记录：

- 缺失图标；
- 标题/按钮/提示语不一致；
- 明显口语化、占位化、歧义化表达；
- 页面可见入口与对应目标页命名不一致。

### 5.2 修补

修补顺序：

1. 路由菜单标题与图标
2. `/portal` 首页快捷入口
3. 各页面主标题与主操作按钮
4. 空状态与提示文案

### 5.3 验证

验证方式：

- 前端 `npm.cmd run build:prod`
- 人工复核菜单标题、图标、空状态和主要按钮文案是否一致

## 6. 风险与取舍

- 风险 1：少量文案调整可能影响既有截图或答辩材料
  - 取舍：仅修明显问题，避免大规模改名。

- 风险 2：图标语义优化带来主观争议
  - 取舍：以“易懂、稳定、与业务匹配”为优先，而不是追求花哨。

- 风险 3：部分页面存在历史遗留入口
  - 取舍：只修当前可见主流程页面，不追求一次清理所有遗留代码。

## 7. 验收标准

- 菜单、快捷入口、主按钮、空状态文案不存在明显缺失；
- 前端整体表达风格基本统一；
- 不引入构建错误；
- 输出一份“发现项 + 已修补项”的简要结果说明。
